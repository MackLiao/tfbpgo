### H2 — `degron` responsive-ratio depends on an unverified `padj` column (silent-wrong OR 500)
- **Where:** `data_prep/src/data_prep/manifests.py:297` (manifest value `('log2FoldChange','padj','','')`); consumed at `backend/internal/api/comparison_topn.go:537-549` (`buildResponsiveExpr`).
- **Why it blocks:** `degron` is in `PRIMARY_DATASETS` and `pertConfigs`, so `?perturbation=degron` reaches `buildResponsiveExpr`, which interpolates `pvalue_col` verbatim into the CASE. The source dict is parity-correct (matches reference `queries.py:18`). But neither gate validates the column exists on the table: startup `db.NewWhitelist` only re-checks `pvalue_col` against `SafeIdentRE` (a regex, not column-existence), and `write_dataset_manifest` validates only `condition_cols` against materialized columns, writing `effect_col`/`pvalue_col` blindly (`manifests.py:403-405,456-466`). The fixture has no `degron` table, so no CI test catches it. Net: if `make data-build` is **not** re-run before cutover, the deployed manifest still says `pvalue_col='pvalue'` and every degron comparison ratio is **silently wrong**; if rebuilt but the materialized table lacks a literal `padj` column, any degron comparison **500s** on `p.padj`. Real user path (Compare Datasets with degron), invisible to CI.
- **Fix:** Add a build-time assertion in `build_duckdb.py` / `write_dataset_manifest` that every non-empty `effect_col`/`pvalue_col`/`log10p_col`/`neglog10p_col` in `DATASET_MEASUREMENT_COLUMNS` exists (via `PRAGMA table_info` / `information_schema.columns`) on its materialized table. Run `make data-build` and confirm `degron` exposes `padj` before cutover. (Not CRITICAL: gated behind the operational rebuild step and the source-of-truth dict is already parity-correct.)

## Medium / Low

### DoS (work amplification)
- **[MEDIUM] `/binding/corr` and `/perturbation/correlations` build an uncapped N-choose-2 UNION-ALL.** `backend/internal/api/binding_corr.go:164-207,215-224`. No pair cap in `serveCorr`/`buildCorrResponse`; the only bound is the CSV length cap = `len(AllDatasets())` (15 binding datasets), and `CheckDataset` does not filter `is_primary`, so all 11 non-primary variants are reachable. C(15,2)=105 branches fused into one `UNION ALL` executed as a single query — exactly the pathology commit `e0f8c92` removed from `comparison/topn`, which has an explicit `maxPairs` cap the corr path lacks. Mitigations verified (size-1 `correlationSemaphore`, 30s `QueryTimeout`, 800MB limit + spill, `ThrottleBacklog`) keep the second pool connection free and prevent OOM, so blast radius is the all-pairs corr feature being serialized/unavailable plus sustained spill — genuine unauthenticated amplification, not a crash. **Fix:** add a pair cap in `serveCorr` symmetric with `comparison_topn.go:171-180` (reject `C(N,2) > maxCorrPairs`, e.g. reuse 12) with a 400 before any DB work.
- **[MEDIUM] `?effect=NaN` / `?pvalue=Inf` on `/comparison/topn` → client-triggerable 500.** `backend/internal/api/validate.go:73-83`. `strconv.ParseFloat("NaN"|"Inf",64)` returns non-finite with `err==nil`, so `parseFloatOr` returns it; it reaches `domain.TopNResponse{EffectThreshold,PValueThreshold}` (plain `float64`, no custom marshaller) and `json.Marshal` fails with `json: unsupported value: NaN` → 500 + ERROR-level log. Fires on the early-return path too, before any DB work. Unauthenticated 5xx-budget / log-flood vector; no data corruption. **Fix:** in `parseFloatOr`, after a successful parse add `if math.IsNaN(v) || math.IsInf(v,0) { return def }`; add a handler test asserting 200/400 not 500.
